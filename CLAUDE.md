## 1. Что это за приложение

Мобильное приложение для записи «тихой охоты» — походов в лес за грибами.
Концептуально похоже на Strava (запись трека тренировки), но главная ценность —
**история найденных грибов**: где, когда, какого типа и сколько.

Три основных раздела:
- **Запись / Record** — запись новой прогулки (трек + отметки грибов/ориентиров + фото)
- **Архив / Archive** — лента прошлых прогулок с детализацией каждой
- **Карта / Map** — агрегированная карта всех находок и маршрутов + статистика

Полное ТЗ — в `SPEC.md`. При любых расхождениях `SPEC.md` — источник истины по
функционалу, этот файл — по архитектуре и правилам разработки.

## 4. Модель данных (Room, 4 таблицы)

Все таблицы создаются в схеме версии 1. С самого начала используем явные
миграции — **никакого `fallbackToDestructiveMigration`**.

### `categories` — категории отмечаемых объектов (в основном грибы)
- `id` (PK)
- `nameKey` — ключ локализации, НЕ готовая строка (для мультиязычности)
- `colorHex` — цвет маркера/точки
- `iconRef` — ссылка на изображение (svg/asset), nullable в базовом варианте
- `order` — порядок в интерфейсе записи
- `isActive` — показывать ли плашку сейчас (редактируется в настройках)

### `walks` — прогулки
- `id` (PK)
- `name` — название прогулки (вводит пользователь)
- `startTime`, `endTime`
- `distanceMeters`
- `avgSpeed`
- `startLat`, `startLon`, `endLat`, `endLon`
- `mushroomCount` — денормализованный счётчик для быстрой ленты архива

### `objects` — найденные объекты (грибы / фото / ориентиры)
- `id` (PK)
- `walkId` (FK → walks)
- `categoryId` (FK → categories)
- `lat`, `lon`
- `timestamp`
- `type` — enum: MUSHROOM / PHOTO / POI
- `photoPath` — nullable, путь к фото (для type=PHOTO)

### `track_points` — точки трека (по аналогии со спортивными трекерами)
- `id` (PK)
- `walkId` (FK → walks)
- `lat`, `lon`
- `timestamp`
- `elevation` — nullable
- `sequence` — порядковый номер точки в треке

**FK с `onDelete = CASCADE`**: удаление прогулки чистит её объекты и точки трека.

## 5. Критичные правила разработки (из ТЗ)

1. **Запись каждого find сразу в БД.** При нажатии «+» тип/место/время гриба
   немедленно коммитятся в Room. При «−» — удаляется последняя отметка этого
   типа. НЕ буферизировать в памяти: если запись прервётся (сбой, закрытие),
   всё успевшее записаться должно сохраниться без планового завершения прогулки.

2. **Локализация через ключи.** Никаких хардкод-строк в UI. Все тексты — через
   механизм i18n (RU + EN минимум, с возможностью добавить языки). Названия
   категорий тоже локализуются через `nameKey`.

3. **Миграции БД с v1.** Структуру закладываем с расчётом на будущие изменения.
   Явные `Migration`, экспорт схемы включён.

4. **Вся логика и UI в `shared`.** Платформенный код — только за `expect`/`actual`.

5. **Иммутабельный UiState + StateFlow** на каждый экран. ViewModel не держит
   Android-зависимостей (общий KMP ViewModel).

6. **Лучшие практики.** Единый репозиторий как точка доступа к данным,
   use-cases для бизнес-логики, DI через Koin, никаких утечек контекста.

**Часть 5 сделана.** Экран «Настройки»: выбор языка (мгновенное переключение

Решения, принятые в Части 1:
- Пакет оставлен как есть (`compose.project.leshy`, сгенерирован визардом) —
  переименование не запрашивалось.
- Домен-модель для таблицы `objects` называется `FieldMark` (тип `MarkType`),
  т.к. слово `object` зарезервировано в Kotlin; Room-сущность — `ObjectEntity`.
- Room 2.8.4 (пакет `androidx.room`), а не Room 3.0 (`androidx.room3`) — 3.0
  вышла совсем недавно, решили не рисковать на самом старте проекта.
- Локализация — через встроенный механизм Compose Multiplatform resources
  (`commonMain/composeResources/values{,-ru}/strings.xml`, `Res.string.*`),
  без сторонних библиотек (moko-resources и т.п.).
- DI границы БД: `expect val platformModule: Module` отдаёт
  `RoomDatabase.Builder<LeshyDatabase>` (Android — через `androidContext()` +
  `getDatabasePath`, iOS — через `NSFileManager` documents dir); общий
  `dataModule` строит БД/DAO/репозитории поверх него.

Решения, принятые в Части 2:
- **MapLibre: реальная карта только на Android, iOS — заглушка.** На этой
  машине нет Xcode/CocoaPods, а `maplibre-compose` тянет нативный MapLibre SDK
  через CocoaPods на iOS-таргете — без Mac эту связку не собрать и не
  проверить. По явному выбору пользователя: `maplibre-compose` подключена
  только в `androidMain.dependencies` (в commonMain её нет вообще), поэтому
  iOS-сборка её не касается. `LiveTrackMap` — `expect/actual @Composable`:
  Android рисует реальную карту (MapLibre + OSM-растровые тайлы без ключей,
  `BaseStyle.Empty` + свой `RasterLayer`, трек как `LineLayer` по `GeoJsonData`,
  находки как `CircleLayer` per цвет категории), iOS — плейсхолдер с текстом
  «появится после сборки через Xcode/CocoaPods». Когда дойдём до Части 4
  (агрегированная карта), нужно будет либо получить доступ к Mac для iOS, либо
  ещё раз явно решить с пользователем, что делать с iOS-картой.
- **Категории засеяны сейчас** (`EnsureDefaultCategoriesUseCase`): 4 гриба
  (белый/лисичка/рыжик/подберёзовик) + служебная неактивная категория
  `category_misc`, на которую вешаются отметки PHOTO/POI (нужна валидная
  FK-ссылка, т.к. `objects.categoryId` не nullable). Список грибов — это
  просто первичный набор; менять/расширять его будет Часть 5 (настройки).
- **GPS** — `LocationTracker` интерфейс + expect/actual через Koin (как БД в
  Части 1). Android: `android.location.LocationManager` напрямую (без
  Google Play Services/Fused Location, чтобы не тянуть лишнюю зависимость).
  iOS: `CLLocationManager` + делегат. Разрешения: Android запрашивает
  `ACCESS_FINE_LOCATION`+`CAMERA` в `MainActivity.onCreate` (единственное
  место, где разрешениям правда нужен Activity); iOS запрашивает через
  системный prompt при первом вызове (авто, никакого доп. кода не нужно).
  Если разрешение не выдано — трек просто не пишется, приложение не падает.
- **Камера** — `expect @Composable fun rememberCameraLauncher`. Android:
  `ActivityResultContracts.TakePicture()` + `FileProvider` (файлы в
  `filesDir/photos`). iOS: `UIImagePickerController` + сохранение в Documents.
- **Use-cases добавлены только сейчас** (Часть 1 их не заводила — там не было
  бизнес-логики). `domain/usecase`: старт/финиш прогулки, запись точки трека
  (+ пересчёт `distanceMeters` по haversine), добавление/удаление отметки
  гриба, добавление фото-отметки, засев дефолтных категорий.
- **RecordViewModel общий между экранами «Запись» и «карта текущей прогулки»**:
  оба `composable<>` в `LeshyNavHost` берут `koinViewModel` с одним и тем же
  `viewModelStoreOwner` (backstack entry экрана `Record`), иначе переход на
  карту создавал бы новый ViewModel и терял активную запись.
- Таймер длительности прогулки — тикает по 1 секунде, не пересчитывается от
  системных часов; расстояние — по реальным дельтам GPS-точек (haversine),
  поэтому не идёт во время паузы.

Решения, принятые в Части 3:
- Добавлена `kotlinx-datetime` (0.8.0) — понадобилась для форматирования
  старта/финиша прогулки (`LocalDateTime.Format` DSL); до этого в проекте
  обходились без неё (Long-таймстемпы + ручной тикер).
- `walks.mushroomCount` (денормализованный счётчик из Части 1) используется
  как есть в карточке ленты — не считаем находки заново из `objects`. На
  детальном экране разбивка «по типам» уже требует агрегации `objects` по
  `categoryId` (только type=MUSHROOM) — это делает `WalkDetailViewModel` через
  `combine()` четырёх Flow (walk/marks/track/categories).
- `WalkMapScreen` переиспользует тот же `expect/actual LiveTrackMap`, что и
  live-карта Части 2, просто с `currentLocation = null` (историческая
  прогулка). Экран «Архив» → «Детали» → «Карта» — тот же паттерн общего
  `koinViewModel` через `viewModelStoreOwner`, что и Record/RecordMap.
- Прогулка без `endTime` (не была явно завершена, напр. после сбоя) не теряет
  данные и всё равно отображается в ленте — длительность в этом случае
  показывается как «—», а не как ошибка.

Решения, принятые в начале Части 4 (iOS-карта):
- **Выбор пользователя: Swift Package Manager вместо CocoaPods**, хотя
  просили именно CocoaPods. Причина: у `maplibre-compose` 0.13.0 в доке
  заявлены оба варианта, но собственное демо-приложение и тесты библиотеки
  используют SPM (плагин `io.github.frankois944.spmForKmp`), плюс CocoaPods
  прекращает поддержку новых версий пакетов в конце 2026 — SPM надёжнее на
  практике. Подтверждено пользователем явным вопросом.
- Подключено через `shared/build.gradle.kts`: плагин `spmForKmp` (версия
  1.9.1 — та же, что использует сам `maplibre-compose` в релизе 0.13.0),
  `iosTarget.swiftPackageConfig { dependency { remotePackageVersion(...) } }`
  тянет `maplibre-gl-native-distribution` (версия `6.25.1`, тоже пиновка
  из `maplibre-compose` v0.13.0), плюс ручные `linkerOpts("-F...", "-rpath", ...)`
  на `binaries.all`, указывающие на scratch-каталог плагина — без этого
  Kotlin/Native framework не находит собранный `MapLibre.xcframework`.
  `gradle.properties`: добавлен `kotlin.mpp.enableCInteropCommonization=true`
  (требование плагина). В `iosApp.xcodeproj` build settings (Debug + Release)
  добавлен `OTHER_LDFLAGS = (-framework Shared, -framework MapLibre)` — порядок
  важен (Kotlin-фреймворк первым), иначе ломается рендеринг текста Compose
  на iOS из-за конфликта символов HarfBuzz (см. предупреждение самой
  библиотеки/JetBrains issue CMP-8882).
- **`LiveTrackMap` перестал быть `expect`/`actual`.** Раз maplibre-compose API
  идентичен на Android и iOS, реализация (Часть 2) перенесена целиком в
  `commonMain` как обычный `@Composable`; платформенные `.android.kt`/`.ios.kt`
  удалены, как и строка-заглушка `record_map_placeholder_ios`. Зависимость
  `maplibre-compose` тоже переехала из `androidMain` в `commonMain`.
- Проверено полностью, включая визуальный рендер: `./gradlew :shared:compileKotlinIosArm64`,
  `:shared:compileKotlinIosSimulatorArm64` и `:shared:compileAndroidMain` — все
  чисто; `xcodebuild` для `iosApp` (симулятор) — `BUILD SUCCEEDED`, линковка
  подхватывает `-framework Shared -framework MapLibre`; `./gradlew :androidApp:assembleDebug`
  тоже проходит. Приложение установлено и запущено в симуляторе (`iPhone 17`,
  iOS 26.5): экран «Запись» открывается без крашей, разрешение на геолокацию
  запрашивается корректно, а экран «Текущая прогулка» реально отрисовывает
  MapLibre-карту с OSM-тайлами и точкой местоположения — то есть то же самое,
  что на Android в Части 2, теперь работает и на iOS.

Решения, принятые в контенте Части 4 (агрегированная карта + статистика):
- **Данные:** `FieldMarkRepository`/`TrackPointRepository` получили `observeAll()`
  (+ `ObjectDao`/`TrackPointDao.observeAll()` без фильтра по `walkId`) — раньше
  оба репозитория умели отдавать данные только по конкретной прогулке.
- **`MapViewModel`** комбинирует 4 Flow (`walks`/`marks`/`trackPoints`/`categories`)
  в сырые данные, затем комбинирует их ещё с `mode` и `selectedPeriod` (свои
  `MutableStateFlow`) — фильтрация по периоду и агрегация статистики целиком
  в `buildUiState()`, без похода в БД повторно при смене фильтра/режима.
- **Период — `MapPeriod(year, month)`**, вычисляется из `walk.startTime` через
  `Instant.toLocalDateTime(TimeZone.currentSystemDefault()).month` (kotlinx-datetime
  0.8.0: `monthNumber` deprecated в пользу `month: Month`, а число месяца — через
  отдельно импортируемое extension-свойство `kotlinx.datetime.number`, не следует
  автоматически из `import kotlinx.datetime.*` на что можно наступить в будущем).
  Список доступных периодов — просто `distinct()` дат прогулок, без панели
  выбора года отдельно от месяца.
- **Подписи периодов — числовые «ММ.ГГГГ»** (`"07.2026"`), без отдельных
  локализованных названий месяцев — так короче на 12 новых строк RU/EN ради
  фильтра, который и так дополняется чекбоксом «За всё время».
- **Карта:** новый `AggregatedFindsMap` (`ui/map/`) — не переиспользует
  `LiveTrackMap` (та работает с одним треком), а рисует один `LineLayer` per
  прогулка (`tracks: Map<Long, List<GeoPoint>>`, приглушённый цвет/прозрачность)
  и один `HeatmapLayer` поверх — по находкам-грибам (`MultiPoint` в
  `GeoJsonData.Features`). Приоритет тепловой карты над маршрутами — по ТЗ.
  Камера один раз прыгает (`cameraState.jumpTo(BoundingBox(...))`) на bounding
  box всех точек трека+находок при первой загрузке данных.
- **Переключатель режимов** — `SingleChoiceSegmentedButtonRow`/`SegmentedButton`
  (Material3), фильтр периода — `FilterChip` в `LazyRow`. Статистика переиспользует
  вид `WalkDetailScreen` (список строк + разбивка по категориям через уже
  существующий `CategoryCount` из `presentation.archive`, чтобы не заводить
  второй одинаковый класс).
- Проверено вживую в симуляторе (iPhone 17, iOS 26.5) на тестовых данных из
  Части 2: тепловая карта реально рисуется поверх OSM-тайлов, переключение
  Карта↔Статистика и фильтр по периоду (「За всё время」↔「07.2026」) работают
  и дают согласованные цифры. Android: `./gradlew :androidApp:assembleDebug`
  проходит чисто (общий Compose-код, реальный запуск на Android-эмуляторе в
  этой сессии не делался — MapLibre на Android уже проверялся вживую в Части 2).

Решения, принятые в Части 5 (Настройки + логотип):
- **Хранение настроек — AndroidX DataStore (Preferences), мультиплатформенно**
  (`datastore-preferences-core` 1.2.1 + Okio), а не Room-таблица и не
  `multiplatform-settings`. Причина: DataStore — единственный из вариантов без
  ритуала «миграция схемы на каждую новую настройку» (ключевое при ожидаемом
  росте числа настроек) и без стороннего non-Google/JetBrains вендора (в
  Части 1 уже был отказ от сторонних либ ради встроенных механизмов — тот же
  принцип). Путь к файлу настроек — `expect/actual platformModule`, тем же
  паттерном, что и `RoomDatabase.Builder` (Android — `filesDir`, iOS —
  `NSFileManager` documents dir).
- **Локализация ПЕРЕСТАЛА идти через Compose Resources `Res.string`.** При
  реализации выяснилось (проверка исходников `components-resources-1.11.1`),
  что `ResourceEnvironment`/`LocalComposeEnvironment`/`LanguageQualifier` в этой
  версии библиотеки — `internal`/`@InternalResourceApi`, то есть переопределить
  локаль в рантайме из кода приложения невозможно; это отменяет решение по i18n
  из Части 1. По явному выбору пользователя заведён свой слой: `i18n/StringKey`
  (enum всех строк) + `i18n/Strings.kt` (`when`-таблицы RU/EN, exhaustive —
  компилятор ловит забытый перевод) + `LocalAppLanguage` (CompositionLocal) +
  composable `stringResource(key)`. Текущий язык хранится в DataStore и
  прокидывается один раз в `App()` через `CompositionLocalProvider`, поэтому
  переключение языка в «Настройках» применяется мгновенно во всём приложении
  без перезапуска. `values/strings.xml` и `values-ru/strings.xml` удалены
  целиком — единственный источник строк теперь `Strings.kt`. Для
  не-composable контекста (например, `RecordViewModel.start()`, который
  раньше звал suspend `getString(Res.string...)`) есть чистая функция
  `string(key, language)`; `RecordViewModel` держит текущий язык в обычной
  var, обновляемой через `collect` в `init` (тот же паттерн, что уже был для
  `categoryRepository.observeActive()`).
- **Набор активных грибов** — переиспользует существующий `isActive` на
  `Category` (никаких новых полей/таблиц); экран «Настройки» — список
  категорий с `Switch`, тумблер сразу пишет через `CategoryRepository.upsert`.
  Служебная неактивная категория `category_misc` (см. Часть 2, нужна для
  FK-ссылки PHOTO/POI-отметок) отфильтрована из списка в `SettingsViewModel` —
  это не гриб, который пользователь должен включать/выключать.
- **Логотип** — по выбору пользователя используется и внутри приложения
  (`composeResources/drawable/leshy_logo.svg`, `painterResource` на экране
  «Настройки»), и как launcher icon на обеих платформах. Рендер SVG→PNG
  сделан локально без интернет-инструментов (`rsvg-convert`/ImageMagick/
  Pillow — ничего не установлено): macOS `qlmanage -t` (Quick Look thumbnail
  generator) как SVG-рендерер + `sips` для ресайза. Три варианта SVG сделаны
  из оригинала: плоский квадрат без скругления (для iOS — Apple сам накладывает
  свою маску, поэтому исходник должен быть full-bleed), круг (Android
  `ic_launcher_round.*`, легаси/API<26), и только сама фигура лешего без фона
  прозрачным — под adaptive-icon foreground. Android adaptive icon: фон стал
  сплошной `#F4F1E8` (`drawable/ic_launcher_background.xml`, один `<path>`),
  foreground переведён с vector drawable на растровый `mipmap/ic_launcher_foreground.png`
  по плотностям (проще и надёжнее, чем вручную конвертировать SVG-обводки/дуги
  circle/ellipse в Android `pathData`). iOS: `app-icon-1024.png` обязан быть
  без alpha-канала (Apple отклоняет иконки с прозрачностью) — исходный рендер
  шёл с альфой, канал убран круглым провозом через JPEG (`sips -s format
  jpeg` → обратно в PNG), визуально не отличимо на растровом лого без
  градиентов.
- Проверено вживую в симуляторе (iPhone 17, iOS 26.5): мгновенное
  переключение RU↔EN во всём приложении (App Bar, нижняя навигация, «Настройки»,
  «Запись» — включая плашки грибов), сохранение выбора языка переживает
  полный перезапуск процесса (`simctl terminate` + `launch`), выключение
  гриба в «Настройках» тут же убирает его плашку на экране «Запись».
  Android: `./gradlew :androidApp:assembleDebug` и `./gradlew build` проходят
  чисто; реальный запуск на Android-эмуляторе в этой сессии не делался (нет
  настроенного AVD) — общий Compose-код тот же, что уже проверен вживую на
  Android в Части 2/4.
- **Найден и исправлен баг навигации** (не Android-специфичный — воспроизведён
  и надёжно отлажен на iOS-симуляторе, три итерации, пока не сошлось всё сразу):
  пока «Настройки» были заглушкой, никто не проверял переход с этого экрана
  обратно на нижнюю навигацию. С реальным контентом выяснилось — после ЛЮБОГО
  захода в «Настройки» кнопка `Record` в `NavigationBarItem` переставала
  работать ОТКУДА УГОДНО (не только напрямую из Settings, но и из Archive/Map
  после того, как Settings хоть раз посещались в этой сессии) — экран
  откатывался обратно на `Settings` вместо `Record`.
  - Итерация 1: `popUpTo(graph.findStartDestination().id)` →
    `popUpTo(graph.id)` — чинило только прямой случай Settings→Record, но не
    полный сценарий (Settings→Archive→Record всё ещё ломался).
  - Итерация 2: убрал `saveState`/`restoreState` целиком (`popUpTo(graph.id)
    { inclusive = true }` без save/restore) — почти правильно, но открыло ДРУГОЙ
    критичный баг: `RecordViewModel` привязан к `viewModelStoreOwner = backStackEntry`
    экрана `Record` (см. Часть 2), и `inclusive = true` уничтожает этот backstack
    entry при каждом уходе на другую вкладку — активная запись прогулки (таймер,
    флаг `isRecording`, накопленная дистанция) полностью сбрасывалась при
    переходе на Archive/Map/Settings и обратно, хотя сама прогулка в БД не
    терялась (только in-memory `RecordUiState`).
  - **Итог (правильная и итоговая версия):** корень обеих проблем —
    асимметрия: кнопка-шестерёнка делала на `Settings` голый `navigate()` без
    `popUpTo`, а нижняя навигация — с `popUpTo(findStartDestination){saveState=true}`
    + `restoreState=true`. Сохранённое-под-route-ключом состояние `Settings`
    (не входящего в нижнюю навигацию, но оказавшегося в том же бэкстеке)
    путало восстановление при последующих переходах. Исправлено — завёл общую
    `NavHostController.navigateToTopLevel(destination)` в `App.kt`, которой
    пользуются И шестерёнка, И все три пункта `NavigationBarItem`:
    `popUpTo(graph.findStartDestination().id) { inclusive = true; saveState = true }`
    + `launchSingleTop = true` + `restoreState = true`. Все четыре
    top-level-экрана (Record/Archive/Map/Settings) теперь проходят через ОДНУ
    и ту же схему pop/save/restore — никакой асимметрии, никакого
    рассинхрона в закэшированном состоянии.
  - Проверено вживую очень подробно, с активной запущенной прогулкой
    (таймер тикал, `Pause`-кнопка): Settings→Record/Archive/Map,
    Settings→Archive→Record (точный сценарий первого бага),
    Record→Map→Settings→Map→Record (полный цикл) — во всех случаях навигация
    работает корректно И состояние записи прогулки (таймер, флаг «идёт
    запись») переживает переключение вкладок без сброса.
- Android: реальный запуск на устройстве/эмуляторе в этой сессии так и не
  делался (нет подключённого девайса и настроенного AVD) — пользователь
  планирует настроить эмулятор отдельно. `./gradlew :androidApp:assembleDebug`
  после финального фикса навигации по-прежнему чистый.

**Баг найден и исправлен позже (первый реальный запуск на Android-эмуляторе,
эмуляторы `Medium_Phone`/`Resizable_Experimental` уже настроены пользователем):**
переход в «Настройки» крашил приложение на Android — `IllegalStateException:
Android platform doesn't support SVG format` (`SettingsScreen.kt:42`,
`painterResource(Res.drawable.leshy_logo)`). Причина — ограничение самой
библиотеки Compose Multiplatform Resources: декодирование SVG у неё
реализовано только через Skia (iOS/Desktop), на Android для растрового
`painterResource` рендерера SVG нет вообще (это отдельная проблема от
растеризации launcher-иконок в этой же Части 5 — там SVG растеризовали из-за
требований самих иконок, а не из-за этого ограничения; в прод-код напрямую
попал ещё и сырой SVG для показа логотипа в UI, и на Android его не
проверяли). Исправлено тем же способом, что и иконки: `leshy_logo.svg` →
растеризован в `leshy_logo.png` (512×512, `qlmanage -t` + `sips`, тот же
пайплайн) и подложен в `composeResources/drawable/` вместо svg — имя ресурса
(`Res.drawable.leshy_logo`) не меняется, код `SettingsScreen.kt` трогать не
пришлось. Проверено вживую на `Medium_Phone`: экран «Настройки» открывается
без крашей, логотип отображается, переключение языка и тумблеры грибов
работают; навигация Record↔Archive↔Map↔Settings проверена повторно — крашей
нет.

**Отдельно замечено при этом: полный `./gradlew build` на этой машине (8 ГБ
RAM) падает с `OutOfMemoryError: Java heap space` в `linkReleaseFrameworkIos*`**
(K2Native-компилятор линкует release-фреймворк iOS с полной оптимизацией —
памяти требует заметно больше, чем обычные `compileKotlinIosArm64`/
`iosSimulatorArm64`, которыми до сих пор и ограничивались проверки iOS).
Не связано с SVG-багом выше. Причина — не нехватка heap самой по себе, а
одновременная конкуренция за RAM с запущенным Android-эмулятором (qemu
занимает ~800 МБ) и параллельной линковкой сразу двух iOS-таргетов
(`iosArm64` + `iosSimulatorArm64`). **Решается точечно, без правки
`gradle.properties`:** остановить эмулятор (`adb emu kill`) и собирать с
`./gradlew build --max-workers=1` (не даёт двум линковкам идти одновременно).
Проверено — с этим `BUILD SUCCESSFUL` за ~4 минуты. Осознанно не стали
поднимать `-Xmx` в `gradle.properties` постоянно: это зарезервировало бы
лишнюю память под каждую обычную Android-сборку, а полная release-линковка
iOS в обычном цикле разработки не нужна — актуальна только перед реальной
сборкой iOS-релиза.

**Найдены и разобраны ещё два бага при первом реальном тестировании карт на
Android (эмулятор + физический Pixel 4a):**

1. **Краш при переходе на экран «Карта» (общая, агрегированная) — исправлен.**
   `IllegalArgumentException: Type of the key MapPeriod(...) is not supported.
   On Android you can only use types which can be stored inside the Bundle.`
   Причина: `MapScreen.kt` использовал `MapPeriod` (обычный `data class`, не
   `Parcelable`/`Serializable`) напрямую как `key` элемента `LazyRow` для
   фильтра периода — Compose требует, чтобы ключ можно было сохранить в
   Android `Bundle`. Исправлено заменой на примитивный `Int`-ключ
   (`it.year * 100 + it.month`).
2. **Размытая подложка карты (растровые OSM-тайлы) — исправлено переходом на
   OpenFreeMap.** Пользователь заметил зернистость/нечёткость подписей на
   карте (и на iOS-симуляторе, и на реальном Android) — маркеры находок и
   линия трека при этом чёткие. Причина: `tile.openstreetmap.org` отдаёт
   растровые PNG-тайлы 256×256 только в обычном (не retina/`@2x`) разрешении
   — на HiDPI-экране они масштабируются вверх, отсюда размытость именно на
   мелких деталях/тексте (маркеры и трек рисуются MapLibre как векторные
   слои поверх, в родном разрешении, поэтому не страдают). Дополнительный
   повод исправить — прямое хотлинк-использование `tile.openstreetmap.org` в
   реальном приложении формально нарушает их usage policy (рассчитана на
   лёгкое/тестовое использование). Решение — переход на бесплатный,
   безключевой векторный стиль **OpenFreeMap** (`ui/map/MapStyle.kt`,
   `BaseStyle.Uri("https://tiles.openfreemap.org/styles/liberty")`), тем
   самым сохраняя изначальный принцип «MapLibre + OSM без API-ключей» из
   раздела 2 этого файла. `RasterLayer`/`rememberRasterSource`/`OSM_TILE_URLS`
   убраны из `LiveTrackMap.kt` и `AggregatedFindsMap.kt` — свои слои (трек,
   маркеры, heatmap) остались как оверлей поверх векторной подложки. Проверено
   вживую на эмуляторе — подписи (`Amphitheatre Parkway` и т.п.) идеально
   чёткие даже при сильном приближении.

**Исправлено: карта текущей прогулки открывалась в точке по умолчанию
(0,0), если открыть её до нажатия «Start».** Причина в обеих платформенных
реализациях `LocationTracker` — они подписывались только на *будущие*
обновления GPS (`LocationManager.requestLocationUpdates` на Android,
`CLLocationManager.startUpdatingLocation()`/делегат на iOS), не отдавая сразу
уже закэшированное системой последнее известное местоположение. Пока не
случалось скачков GPS-провайдера, `currentLocation` в `RecordUiState`
оставался `null`, и `LiveTrackMap` показывал дефолтную точку. Это не зависело
от старта прогулки как такового — `locationTracker.track()` в
`RecordViewModel.init` слушает GPS независимо от `isRecording`, поэтому баг
проявлялся именно в задержке первого фикса. Исправлено эмитом кэшированной
точки сразу при подписке: Android —
`locationManager.getLastKnownLocation(provider)`, iOS — `manager.location`
(кэшированное свойство `CLLocationManager`, доступно без ожидания callback'а
делегата). Проверено вживую на эмуляторе с зафиксированной mock-геолокацией
(`adb emu geo fix`): открыл карту записи сразу после запуска приложения,
не нажимая «Start» — карта сразу показывает реальную точку, а не (0,0).

**Найден и исправлен ещё один краш навигации — воспроизводился и на Android,
и на iOS.** Симптом: если открыть карту текущей записываемой прогулки (кнопка
«View map» на экране «Запись», неважно, идёт запись или на паузе) и затем
нажать любую из нижних вкладок (Запись/Архив/Карта) — приложение падало.
Причина: `LeshyNavHost.kt` в `composable<Destination.RecordMap>` вызывает
`navController.getBackStackEntry(Destination.Record)` при каждой
рекомпозиции — чтобы взять тот же `viewModelStoreOwner`, что и у экрана
`Record` (см. Часть 2). Переход по нижней вкладке использует
`navigateToTopLevel` → `popUpTo(graph.findStartDestination().id)
{ inclusive = true; saveState = true }` (см. баг-фикс навигации в Части 5) —
это выталкивает запись `Record` из бэкстека СИНХРОННО, до того как успевает
закончиться recomposition уходящего экрана `RecordMap` во время
exit-анимации перехода. На этом кадре `getBackStackEntry(Destination.Record)`
бросает `IllegalArgumentException: No destination with route ... Record is
on the NavController's back stack` — краш. Тот же паттерн (без учёта того же
случая) был и в `composable<Destination.WalkMap>`
(`getBackStackEntry(Destination.WalkDetail(...))`) — тоже исправлен
превентивно, хотя пользователь этот путь ещё не проверял. Исправлено оборачиванием
обоих вызовов в `runCatching { ... }.getOrNull()` — если родительская запись
уже пропала (переход в процессе), composable просто ничего не рендерит на
этот короткий кадр вместо краша (экран всё равно анимированно исчезает).
Проверено вживую на эмуляторе длинной цепочкой переходов
(Запись→КартаЗаписи→Настройки→Запись→Архив→Детали→КартаПрогулки→Запись) —
крашей нет; экран `Record`/`RecordMap` корректно восстанавливается через
`restoreState=true` (последнее посещённое состояние внутри вкладки).

**Известное ограничение (НЕ исправлено, оставлено как документированный
баг окружения): `HeatmapLayer` крашит нативную библиотеку MapLibre
(`libmaplibre.so`) — но только на Android-эмуляторе, не на реальных
устройствах.** Симптом: при открытии экрана «Карта» (там, где рисуется
`AggregatedFindsMap` с `HeatmapLayer` поверх находок) — `Fatal signal 11
(SIGSEGV)` в `RenderThread`, всегда один и тот же адрес сбоя и офсет внутри
`libmaplibre.so`, независимо от данных (проверено на вырожденных
одноточечных и на разнесённых координатах) и от GPU-режима эмулятора
(`hw.gpu.mode=auto` и принудительный `swiftshader_indirect` — одинаково
падает). Изначальная гипотеза (несовместимость с 16KB page size системного
образа `google_apis_playstore_ps16k`) **опровергнута** — краш воспроизведён
и на специально созданном тестовом AVD `Pixel_Test_4k` с обычным (4KB,
`system-images;android-36;google_apis_playstore;arm64-v8a`) образом, тем же
самым адресом сбоя. При этом **на реальном физическом Pixel 4a пользователя
этот же экран открывается без ошибок** — то есть краш специфичен именно для
графического стека Android-эмулятора (вероятно, трансляция ANGLE/SwiftShader
→ Metal на macOS-хосте в связке с нативным heatmap-рендерингом конкретно этой
версии `libmaplibre.so`/`maplibre-compose`), а не баг кода проекта и не угроза
реальным пользователям. Осознанно не стали городить defensive-код
(runtime-детект окружения + fallback на `CircleLayer` вместо `HeatmapLayer`) —
раз реальное устройство работает нормально, это не блокер; просто нужно
помнить, что экран «Карта» (в отличие от остальных карт в приложении) нельзя
доверять при тестировании на Android-эмуляторе на этой машине — только на
реальном устройстве или на эмуляторе другого хоста/конфигурации.
